기존의 문제는 무엇일까

저 자신이 사용자로서 사이트들을 이용할 때 불편했던 사이트들을 적지 않게 봐왔던 개인적 경험을 토대로, 웹 사이트를 이용함에 생기는 문제점들을 추려본 결과, 제가 정리한 문제점들은 지나치게 높고 부정적 경험을 초래하는 상품 광고 빈도, 직관적이지 않은 UI 디자인, 너무 많은 스크롤 횟수, 목적성이 뚜렷하지 않은 배치 등 모두 MD 관련 혹은 UI / UX 두 가지의 영역에서 발생한 웹 사용자의 부정적 경험 초래가 문제라고 판단하였습니다.

그 문제는 왜 발생했는가

신상품이나 프로모션 상품들의 경우 노출을 통해 클릭을 유도해야 하고, 높은 클릭률이 곧 판매와도 높은 연관이 있는 부분이긴 하나, 이에 따라 UI / UX 디자인이 무너진다면 이는 곧 사용자 경험에 있어서 치명적인 결함을 초래하게 됩니다. 더불어 환기 없이 진행되는 긴 스크롤, 목적성이 뚜렷하지 않은 배치 혹은 직관적이지 않은 UI 디자인은 사용자에게 지루함을 느끼게 하고, 웹을 이용하는 데 있어서 부정적인 감정을 심어줄 수 있고, 이는 곧 사이트 이탈로 이어지게 될 가능성이 높습니다.

데스크탑의 경우, 상단 네비게이션은 항상 따라다니게 되며, 스크롤 즉시 반투명 상태로 변하다가, 마우스가 올려지는 순간 흰색을 띕니다. 마우스가 올라가 있으면 네비게이션에서 무엇을 찾고 있다고 판단해서 가독성을 높이기 위함입니다.

검색창을 누르고, nav li에 hover했을 때 생기는 변화입니다. 네비게이션 속 메뉴들을 숨겨놨다가 사용자 동작에 반응하여 보이는 설계로 공간 확보와 가독성을 높이는 효과를 가져올 수 있었습니다.

판매 카테고리의 또 하나의 서브 카테고리가 더 있는 경우는, 네비게이션을 누르면 또 하나의 다른 네비게이션 modal이 animate되면서 등장하게 구현하였습니다. 이로 인해 서브 구조가 많음에도 네비게이션의 일관적이고 간단한 디자인을 유지할 수 있었습니다.

모바일은 간단한 UI를 버튼 형식으로 제공합니다. 아이콘은 직관적이며, text도 함께 포함하고 있습니다. 이의 간단한 조작을 통해 심플한 메뉴로 이동하게 되며, 이곳을 통해 메인페이지에서 다른 경로로 이동이 바로 가능하도록 설계했습니다.

또 모바일의 상단과 하단 Nav는 fixed되어 따라오게 되는데, 사용자의 스크롤에 반응하여 화면에서 사라집니다. 이는 모바일 기기의 크기를 고려한 의도적 설계이며, 스크롤을 미세하게 위로 올리는 즉시 다시 나타나게 됩니다. 이를 통해 모바일 기기에서 화면에 보이는 브라우저를 최대한 확보합니다.

판매 카테고리의 메인 페이지입니다. 다른 판매 카테고리 내의 메인 페이지로 이동이 가능하게 끔 sub directory를 배너에 자연스럽게 float 해놓았으며, 실제 프로덕트 한매 페이지로 이동을 위해 배너 밑에 곧이어 gnb도 배치하였습니다. 해당 페이지의 모바일의 경우엔, float된 sub directory는 과감하게 제거하고, 모바일 기기의 맞춘 방식으로 바꿨으며, 해당 설명은 밑 이미지에서 이어집니다.

모바일의 터치 방식을 적극적으로 이용해 상단의 메뉴bar를 탭 하면 다른 메인 페이지로 이동이 가능하게끔 설계해 놓았습니다. 또 화살표 방향을 아래로 배치하는 것을 통해, 누르면 무엇인가 내려올 것이라는 암시를 합니다.

Sort Filter UI도 직관적으로 배치해 놓았습니다. 사용자가 버튼의 의도가 무엇인지 정확하게 파악하고, 눌렀을 때 무슨 기능이 실행될 지 단번에 알아볼 수 있도록 의도하였습니다.

브라우저 크기에 따른 resize 이벤트를 통해 새로고침을 넣는 것은 반응형을 고려해 봤을 때 스크립트와 연관하여 중요하다 생각하는데요, 기존에는 resize시 if문을 통한 새로고침으로 코드를 작성하였으나, 모바일 환경에서 스크롤에 따라 계속 새로고침이 되는 문제를 발견했었습니다. 처음엔 이유를 몰라 많이 당황했으나, 서치를 통해 원인을 찾았고, 해결까지 하게 되었습니다.

본인 블로그 링크 - 모바일 resize 새로고침

우선 resize에 delay를 부여하지 않고 새로고침 코드를 넣게 되면, 브라우저 과부화가 너무 심해지기 때문에 delay를 넣었습니다. 기존 코드에서 모바일 스크롤 시 새로고침이 되는 문제점은, 모바일은 하단 주소창으로 인해 height값에 변화가 계속 생기기 때문에 스크립트 코드가 계속 발생했던 것인데요, 이는 곧 브라우저의 크기 변화를 탐지하는 것에서, width값의 변화를 탐지하는 것으로 해결하였습니다.

문제5. 브라우저 로딩에 따른 프리로더

사이트에 로드될 요소가 많으면 로딩하는데 시간이 소요될 수 있습니다. 사용자가 모든게 로드되기 전에 급하게 스크롤 할 경우, 놓치는 컨텐츠가 있을 수 있고, 이는 클릭률의 저하로 이어질 가능성이 높습니다. 이 문제를 해결하기 위해 알아보던 중, 프리-로더 기능을 발견, 적용하게 되었습니다.

본인 블로그 링크 - pre-loader

제이쿼리를 이용하면 쉽게 구현이 가능했습니다. 도큐먼트가 로딩되는 동안, 모든 요소 위에 스피너가 돌아갈 pre-loader div를 하나 띄워놓고, 바디 스크롤을 막아 놓은 후, 브라우저가 로딩되고 있다는 것을 알려주기 위해 svg를 증감식과 rotate, 그리고 setInterval을 이용해 계속해서 돌아가게끔 만들었습니다. 그 이유는 로딩 시간을 그냥 기다리게 만든다면, 이는 곧 이탈률의 증대로 연계될 가능성이 높기 때문입니다. 단순히 화면 가운데에서 스피너가 돌아가는 것만으로 잠깐의 이목을 끄는 것만으로도, 사용자의 이탈을 막는 효과를 기대할 수 있습니다. 그 후 브라우저가 로딩되면 body에 loaded 클래스를 부여해 스크롤을 가능하게 만들고, pre-loader div를 제거하는 함수를 실행시킵니다. 사용자는 이로 인해 브라우저가 완전히 로딩된 후 탐색할 수 있게 되고, 버벅임 없이 깔끔한 사용 환경을 제공하게 됩니다.

문제6. 설명페이지 제작에서의 문제

이 설명 페이지를 작성하면서 발생한 문제입니다. 위에서 서술한 스크롤에 따른 이벤트 함수를 이 페이지에서도 작성하였는데요, 페이지 설계상, 우측에 따라다니는 TOC(목차)를 이용하고 있으며, 목차 script는 제이쿼리 플러그인을 사용할수도 있었지만, 검색을 통해 발견한 바닐라 js 코드를 채용했습니다. 그리고, 목차 외에도 4개의 챕터가 준비되어 있고 각 챕터를 클릭하면 그 챕터만 보여지는 식으로 만들었으며, 각 챕터엔 목차가 배치되어 있는 형태로 구성하는 과정에서 발생한 문제를 서술하겠습니다.

본인 블로그 링크 - 설명 페이지 제작 문제

우측에 위치한 목차를 TOC라고 부르는데, 이 toc는 h2 h3 h4 태그를 읽어, 변수에 담아둔 빈 div에 ul과 li, a로 담는 스크립트입니다. 여기서 챕터가 4개이기 때문에, 각 각 h2 h3 h4가 챕터마다 다른데, 이게 목차에 전부 생겨버리는 첫번째 문제가 발생했습니다.

이를 해결하기 위해, container 안에서 각 챕터 section을 나누고, 각 목차를 변수에 담아서 함수하나로 재활용하는 식으로 스크립트를 개조했습니다. 이에 따라 각 챕터에 맞게 목차가 h2 h3 h4를 포함시키는 것은 성공했으나, 이 챕터를 하나씩 보이게 하는 과정에서 두번째 문제가 발생했습니다.

해당 스크립트는 제가 작성한 스크롤 이벤트 코드인데요, 스크롤 값과 요소의 위치의 계산식을 이용하고 있는 코드인데, 각 챕터를 display: none 하고 block 처리 하는 방법으로 챕터를 보여줬다 숨겼다 하는 식으로 하게 되니, 첫 챕터를 제외한 다른 챕터에는 hide되어 있어야 할 요소들이 모두 보여져 있는 상태가 되어 있었습니다.

따라서 이를 해결하기 위해, 각 챕터를 absolute로 띄우고, opacity를 0으로 맞춰논 이후, 브라우저의 left로 무한히 빼낸 다음, bottom으로 무한히 밑에 배치해 놓았습니다. 이로 인해, 한 챕터를 읽을 때 다른 챕터들은 무한히 아래에 있기 때문에, 계산식에 위배되지 않게 됩니다. 그러나 여기서 세번째 문제가 발생했습니다. 다음 챕터를 클릭하면 아래에 박혀 있는 클릭한 챕터를 빼내오고, 읽고 있는 챕터를 아래로 넣는 과정이 심미적으로 옳지 않다고 느꼈고, 이를 해결하기 위해 고민하던 중, 제작 기간동안 봤던 연극을 떠올리게 되었습니다.

연극에서 다음 챕터를 준비하기 위해 불이 꺼지고, 커튼이 쳐진 후 다음 무대를 준비합니다. 그 점을 정확히 똑같이 이 페이지에 구현해 놓았습니다.

먼저 브라우저 왼쪽에 커텐을 하나 숨겨 놓고, 각 챕터를 배열에 담습니다. 사용자가 챕터를 클릭하게 되면, 이 챕터가 기존에 나와있는 챕터인지 분별하고, 기존에 나와있는 챕터라면 return값을, 다른 챕터라면 숨겨뒀던 커텐을 오른쪽으로 펼치는 키프레임을 실행시킵니다. 커튼이 쳐지고 나면, 기존 챕터를 아래로 보내버리고, 챕터의 썸네일로 볼 수 있는 상단 gnb의 인덱스값과 같은 인덱스값의 챕터를 위로 뽑아 옵니다. 이 모든 과정 이후에, 커텐이 다시 펼쳐지도록 설계해 놓았고, 이 방식을 통해서 만들어 둔 스크롤 이벤트와의 호환이 완벽하게 되는 것을 확인할 수 있습니다.

레이아웃의 일관성

Select More. 에서는 메인페이지에서 일관성을 유지하기 위해 desktop 기준 width: 1562px을 잡아 놓고, 양쪽 margin에 auto를 줘 고정된 수치 안에서 작업을 하였습니다. 이렇게 계산된 의도적인 px수치는 4k나 여러 기기 환경에서 일관된 layout을 유지하기 쉬우며, 내부 요소 또한 고정된 px값 안에서의 일률적인 간격들을 주어 layout 일관성을 가져올 수 있습니다. desktop을 벗어나 mobile이나 tablet 환경에서는, 고정된 px수치에서 main-container의 width를 %값으로 다시 바꾼 후, 동일하게 가로 margin auto를 주어 어떤 기기에서든 80%의 일괄적인 여백 레이아웃을 확보하였습니다.

반응형에 따른 가로-세로 구조

데스크탑과 모바일, 태블릿은 기기 특성이 모두 다릅니다. 동시에, 기기를 조작하는 느낌또한 다르며, 가로 세로 비율마저 다릅니다. 그렇기 떄문에, 이를 감안한 레이아웃의 조정은 반응형 웹에서는 선택이 아닌 필수인데요, Select More. 에서는 미디어쿼리를 이용해서 각 기기 특성에 맞게, display Flex 속성에서 세로배치 , 가로배치가 된다거나, 2줄로 진열된 것이 1열로 진열되다가도, 다시 구조가 바뀌는 등, 반응형 최적화에 힘썼습니다. 또한, product page에서는 display flex 속성과 width에 percentage값 + box-sizing border box + padding을 활용해서, product list가 기기 크기에 따라 자연스럽게 자동 정렬되도록 코드 구성을 하며, 전체적인 반응형에 크게 신경을 썼습니다.
